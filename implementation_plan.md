# Architectural Refactor: FastAPI, Ngrok, and Geometry Generation

This document outlines the planning and analysis for two major architectural decisions for the SmartHVAC Studio FYP.

## 1. Replacing Firebase with FastAPI + Ngrok

### Concept
Eliminate Firebase as the middleman. Instead, run a `FastAPI` server directly inside the Google Colab environment and expose it to the public internet using `ngrok` (or `pinggy`). The Vanilla JS frontend will send HTTP POST requests directly to this temporary URL.

### Pros
- **Zero Cloud Configuration:** Eliminates the need for Firebase projects, Firestore databases, and API keys.
- **Industry Standard:** Transitioning to a REST API architecture is a highly professional and standard approach.
- **Security:** Colab acts as an isolated sandbox, and the tunnel is fully encrypted.

### Implementation Steps
1. **Backend (Colab/Python):**
   - Create a `FastAPI` application with an endpoint (e.g., `/api/simulate`).
   - Implement `CORS` middleware to allow requests from the web frontend.
   - Create Pydantic models to accept the user's prompt and simulation settings.
   - Integrate `pyngrok` (or `pinggy`) to spin up the public tunnel and print the URL to the console.
2. **Frontend (Vanilla JS):**
   - Remove all Firebase SDKs (`firebase-app.js`, `firebase-firestore.js`).
   - Add a configuration input field in the UI where the user pastes the temporary Ngrok URL provided by Colab.
   - Replace Firebase Firestore listeners with asynchronous `fetch()` API calls to the Ngrok URL.

## 2. Geometry Engine: Custom Logic vs. OpenStudio vs. Topologicpy

### The Options
We have three potential paths for handling the complex building geometry generated by the LLM:

1. **Custom IDF Generator + Plotly (Current Approach)**
   - **Pros:** It is already built. We mathematically verified it works. The Plotly visualizer allows for instant, interactive 3D rendering right in the web browser, giving the project a huge "Wow Factor" (zero-install requirement for evaluators).
   - **Cons:** Manual coordinate math is error-prone. If you add angled walls, L-shaped rooms, or overlapping stories later, the math becomes extremely complex to maintain.

2. **OpenStudio Application & Python SDK**
   - **Pros:** OpenStudio is the industry standard for BEM. It handles adjacencies and HVAC logic flawlessly.
   - **Cons:** OpenStudio's GUI is a Heavy Desktop Application. If we rely on it for visualization, the evaluator has to download an `.osm` file and open it locally. This breaks the seamless web experience. Rewriting the backend to use the OpenStudio SDK is also a heavy lift.

3. **Topologicpy (New Discovery)**
   - **What it is:** A powerful Python library specifically designed for architectural topology and Building Energy Modeling (BEM). It builds non-manifold geometries (CellComplexes) which mathematically guarantee that adjacent rooms share clean faces without gaps or overlaps.
   - **Pros:** 
     - It solves the "overlapping rooms" and "adjacency" math automatically. You just define the rooms, and it figures out the shared walls.
     - It is highly optimized and runs completely inside Python (Colab).
     - It can export cleanly to EnergyPlus/OpenStudio.
   - **Cons:** It has a learning curve. We would have to completely rewrite `geometry_util.py` to use `topologicpy` classes instead of calculating vertices manually.

### My Recommendation for the FYP

1. **Visualization: DO NOT drop Plotly.** 
   The ability to type a prompt and instantly see a 3D building render in a browser is incredibly impressive for an FYP. You should keep Plotly on the frontend no matter what engine you use on the backend.

2. **Backend Engine: Topologicpy is a Fantastic Upgrade!** 
   Since `topologicpy` operates entirely in Python, we can use it in your Colab backend to replace `geometry_util.py`. 
   - **How it would work:** The AI outputs dimensions. We feed those dimensions to `topologicpy` to generate `Cells`. We merge them into a `CellComplex`. Topologicpy automatically handles all the shared walls. Then, we write a small script to export that Topologic geometry into the raw EnergyPlus IDF format, which gets sent to the frontend for Plotly to render.
   - **Why this is smart:** It makes your geometry engine mathematically bulletproof (you won't have to debug X/Y/Z origins anymore), while preserving the beautiful web-based Plotly visualizer.
